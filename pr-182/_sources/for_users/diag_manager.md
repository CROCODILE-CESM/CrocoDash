# Managing Diagnostics (case.manage_diags)

MOM6 diagnostics are controlled via a `diag_table` file that specifies which variables to output, at what frequency, and in what format. CrocoDash provides `case.manage_diags` to give you an interactive UI for editing this file without having to hand-edit a complex text format. You can see the official diag_table docs (they are very good) here: https://mom6.readthedocs.io/en/dev-gfdl/api/generated/pages/Diagnostics.html

## Overview

`case.manage_diags` forks the [`mom6_diagnostics_gui`](https://github.com/anthony-meza/mom6_diagnostics_gui) package to provide a Jupyter widget UI for:

- Viewing all diagnostics currently enabled in your `diag_table` (Must have one either in `SourceMods/src.mom/`, which means a diag_table you have edited, or `CaseDocs/`, which is generated by running `./preview_namelists`)
- Adding new diagnostics from a list of available ones (if an available_diags file exists in your run directory)
- Removing or modifying existing diagnostic entries
- Writing the updated `diag_table` to your case's `SourceMods/src.mom/` directory

## Usage

```python
ui = case.manage_diags()
display(ui)
```

Optionally, provide a path to an `available_diags` file listing all diagnostics MOM6 can output for your configuration:

```python
ui = case.manage_diags(path_to_available_diags="/path/to/available_diags.000000")
```

## How It Works

1. **Finds a starting `diag_table`**: CrocoDash looks in this order:
   - `<caseroot>/SourceMods/src.mom/diag_table` (your previously customized version)
   - `<caseroot>/CaseDocs/diag_table` (the CESM-generated default or picked up from SourceMods)
   - Runs `./preview_namelists --component ocn` to generate one if neither exists

2. **Finds available diagnostics**: If `path_to_available_diags` is not given, it looks for `available_diags.000000` in your run directory. This file is generated by MOM6 on the first time-step of a run. If it doesn't exist yet, the UI will still work but won't be able to suggest new diagnostics to add.

3. **Launches the UI**: A Jupyter widget is returned and stored as `case.ui`. Any changes you make in the UI are written to `<caseroot>/SourceMods/src.mom/diag_table`, which overrides the default for your case.

## Getting the Available Diags File

MOM6 only writes `available_diags.000000` after running for at least one time-step. A typical workflow is:

1. Build and run your case briefly (e.g., a 1-day test run)
2. Find `available_diags.000000` in your run directory
3. Call `case.manage_diags()` — it will pick it up automatically

```python
# After a short test run, available_diags.000000 will exist in RUNDIR
ui = case.manage_diags()
display(ui)
```

## Output Location

The edited `diag_table` is saved to:

```
<caseroot>/SourceMods/src.mom/diag_table
```

CESM picks up files in `SourceMods/src.mom/` automatically when building or running, so no additional steps are needed to apply your changes.

## Notes

- `manage_diags` must be called from a Jupyter environment to display the widget.
- The UI is also stored as `case.ui` if you need to access it later in your notebook.
- If no `diag_table` can be found and `preview_namelists` also fails, a `ValueError` is raised.
